## what
- エラーを適切に処理して開発者とユーザーに知らせること
	- エラーは「予期されるエラー」と「予期しないエラー」の大きくふたつに分けられる
		- 予期されるエラー（Expected）: ビジネス上起こりうる失敗。メールアドレスの重複、在庫切れ、etc...
		- 予期しないエラー（Unexpected）: プログラムのバグやインフラ障害。DB接続切れ、APIエラー、etc...
- エラーハンドリングの３大原則
	- 握りつぶさない
	- 適切な層で処理する
	- ユーザーと開発者で情報を分ける
	- 例
		- DB層:   「接続タイムアウトした」 → 自分では判断できない → 上に伝える
		- Service層: 「DBが落ちてるな」 → Sentryに通知 → 上にはビジネスエラーとして伝える
		- API層:   「内部エラーだな」 → HTTPステータス500で返す
		- UI層:    「エラーが来た」 → ユーザーに "しばらくお待ちください" と表示
- エラーの上げ方
	- throw / try-catch
	- Result型
		- 例外が起こりうることが型として明示されるため、エラーハンドリングを実装してないとコンパイルエラーになる。[[関数型プログラミング]]のアプローチ。モダンなシステムではこの方法を使う。
- フロントエンドのエラー表示戦略

| 表示手段                   | 用途               | 例                       |
| ---------------------- | ---------------- | ----------------------- |
| Toast通知                | 一時的なエラー。操作は継続可能。 | 「保存に失敗しました」             |
| フォームバリデーション            | 入力ミス。            | 「メールアドレスの形式が不正です」       |
| エラーページ(Error Boundary) | 画面全体が壊れた時        | 「問題が発生しました。再読み込みしてください」 |
| リダイレクト                 | 認証切れなど           | →ログイン画面へ                |
	- `alert()`はUXが悪いのでモダンなアプリでは避けられる。
- Observability
	- [[Sentry]]などの監視ツールへは、Unexpected系だけを通知する
