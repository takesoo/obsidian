## コンセプト

- Bandcampで購入した音源も、CDから取り込んだ音源も、**1つのライブラリとしてまとめて管理・再生**したい。
    
- PCとスマホの両方で**同じライブラリを共有**したい。
    
- 家でも外でも（場所を問わず）**ストリーミング再生**したい。
    
- 既存サービス（Bandcamp公式アプリ / Spotify / Apple Music）では満たせないニッチを、自分用にまず作る。
    

ざっくり言えば：

> 「自分のためのSpotify」＝ 手持ち音源専用のパーソナル音楽クラウド

## 現状の不満・課題感

- Bandcamp：
    
    - Bandcampで買った音源はBandcampアプリでしかライブラリ管理できない。
        
    - CDリッピング音源や他の購入音源はBandcampでは扱えない。
        
- ローカルプレイヤー（iTunes / foobar2000 など）：
    
    - PCとスマホでライブラリを同期する仕組みは自前で頑張る必要がある。
        
    - 外出先ではPCを立ち上げるか、別途転送が必要でだるい。
        
- Spotify / Apple Music 等：
    
    - 「自分が買ったファイルをクラウドにアップする」用途には弱い or 想定されていない。
        

## やりたいこと（要件）

### ユースケース

1. PCにある音源（Bandcampダウンロード + CDリッピング）を**まとめてアップロード**したい。
    
2. PC・スマホの両方から**同じライブラリ**を見たい。
    
3. どこにいても（自宅 Wi‑Fi / 外の4G/5G）**ストリーミング再生**したい。
    
4. 最低限：
    
    - 曲一覧
        
    - アルバム一覧
        
    - アーティスト一覧
        
    - シンプルなプレイリスト
        
    - シャッフル / リピート
        

### MVPの割り切り

- **対象ユーザー：自分のみ（シングルテナント想定でOK）**
    
- 認証は超シンプル（最低限のログイン or 自分専用トークン）
    
- UIは凝りすぎない（Spotifyクローンを目指さない）
    
- オフライン再生は一旦後回し（まずはストリーミングだけ）
    
- モバイルは最初はPWA or Webでもよい（ネイティブアプリは次のステップ）
    

## アーキテクチャのざっくり案

### 構成イメージ

- フロントエンド：Next.js（Webプレイヤー）
    
- バックエンド：Next.js API Routes or Edge Functions
    
- ストレージ：S3（音源ファイル）＋ CloudFront（配信）
    
- DB：PostgreSQL（Supabaseでも可）
    
- 認証：自分用なので最初は最低限（Basic Auth / simple auth）
    

### 流れ（MVP）

1. PCブラウザでWebアプリを開く。
    
2. `ドラッグ&ドロップ` で音源フォルダをアップロード。
    
3. API が S3 にアップロードしつつ、メタデータ（曲名・アルバム名・アーティスト・トラック長など）をDBへ保存。
    
4. Webプレイヤー上で：
    
    - アーティスト一覧 / アルバム一覧 / 曲一覧を表示。
        
    - クリックで再生（`/api/stream?trackId=xxx` → S3の署名付きURL or CloudFrontにリダイレクト）。
        
5. スマホからも同じWebアプリにアクセスして、同じライブラリを再生。
    

## 技術メモ

### データモデル（ざっくり）

- `tracks`
    
    - id
        
    - title
        
    - album_id
        
    - artist_id
        
    - duration
        
    - file_path（S3キー）
        
    - format（mp3/flac/wav 等）
        
- `albums`
    
    - id
        
    - title
        
    - artist_id
        
    - year
        
    - cover_image_path
        
- `artists`
    
    - id
        
    - name
        
- `playlists`
    
    - id
        
    - name
        
- `playlist_tracks`
    
    - playlist_id
        
    - track_id
        
    - order
        

※ 最初は「ユーザーID」は1固定でもよい（完全自分専用サービス前提）。

### 再生方法

- Web：
    
    - HTML5 Audio（`  `）で十分。
        
    - 曲単位のAPIからstream URLを取得。
        
- 音源フォーマット：
    
    - MVPではとりあえず mp3 に限定してもよい。
        
    - 余力があれば、アップロード時に ffmpeg でmp3に自動変換してもよい。

**競合ざっくり比較（あなたの文脈向け）** 

**Apple Music**

- **特徴／強み**

- iCloudミュージックライブラリで、自前音源（CDリップ・Bandcamp購入など）もクラウド化して全デバイスから再生可能。
- ロスレス／ハイレゾ／空間オーディオが追加料金なし。
- iPhone・Mac・Apple Watch・CarPlayなどApple製品との統合が強い。
- 邦楽・J‑POPのカタログが比較的強い。

- **弱み／不評ポイント**

- 無料プランなし（常に有料サブスク前提）。
- レコメンドが保守的・刺さりにくいという声。
- Android・Windowsでの体験が微妙（アプリが重い、UIが古いなど）。
- iCloudミュージックライブラリの同期でライブラリが崩れる／タグ書き換えなどのトラブル事例。

**Spotify**

- **特徴／強み**

- Discover Weekly / Release Radar 等、レコメンドと自動プレイリストの評価が非常に高い。
- UIが軽く、マルチデバイス・外部連携（ゲーム機、Discordなど）が広い。
- 広告付きの無料プランが強力（2025年以降はスマホでもかなり自由にオンデマンド再生可）。
- 「新しい曲・アーティストの発見」に最適、ソーシャル機能も豊富。

- **弱み／不評ポイント**

- ローカルファイル／自前音源との統合は中途半端（Apple Musicほど“クラウド一元管理”できない）。
- 高音質志向ではApple/Amazonに劣ると言われがち（ロスレス対応は最近）。
- 無料版は広告・音質・オフライン再生で制約が多い。

**差別化軸（あなたのプロダクト視点で使えそうなポイント）** 

- **自前ライブラリ統合の質**

- Apple: 強いがブラックボックス＋Apple縛り＋同期トラブルリスク。
- Spotify: 発見は強いが、自前音源クラウドとしては弱い。

- **プラットフォームの開放度**

- Apple: Appleデバイス前提。
- Spotify: クロスプラットフォームだがローカル音源統合は限定的。

- **レコメンド哲学**

- Apple: “自分のライブラリ中心の整理・所有”寄り。
- Spotify: “パーソナライズされた発見・プレイリスト”寄り。

ここを踏まえると、  
「マルチプラットフォームで、自前ライブラリを安全に一元クラウド化しつつ、Spotify級の発見体験を乗せる」  
のが既存2社との一番わかりやすい差別化になりそうです。
