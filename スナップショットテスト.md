## what
- テスト対象のモジュールにインプットパターンを与え、アウトプットをシリアライズしたものを保存しておき、変更の前後で変化していないことを保証するテスト。
	- フロントエンドの場合、テスト対象のコンポーネントのレンダリング結果をスナップショットとして保存しておき、テスト実行時にスナップショットとレンダリング結果を比較することでテストを行う。
1. 初回実行：出力結果をテキストファイル(`xxx.test.js.snap`)としてローカルに保存する
2. 2回目以降：出力結果をスナップショットと比較し、一致しなければエラーとする
- 実装コストが低いというメリットがある反面、テストが壊れやすいというデメリットもある
## why
- [[単体テスト|Unit test]]に比べて楽に実装できる。出力の仕様が変わるとスナップショットを作り直す必要があるので、仕様変更の多い箇所には不向き。費用対効果の低い箇所ならマッチする場合もある。
## how
### Jestの場合
```ts
import { render } from '@testing-library/react';
import MyButton from './MyButton';

test('ボタンが正しくレンダリングされること', () => {
  const { container } = render(<MyButton label="送信" />);
  
  // これだけで、出力されたHTMLの「スナップショット」を比較してくれる
  expect(container.firstChild).toMatchSnapshot();
});
```
### Vitestの場合
```ts
// Button.test.tsx
import { describe, it, expect } from 'vitest';
import { render } from '@testing-library/react';
import React from 'react';
import { Button } from './Button';

describe('Button コンポーネント', () => {
  it('正しくレンダリングされ、スナップショットと一致すること', () => {
    // 1. コンポーネントを仮想DOMにレンダリング
    const { container } = render(<Button label="クリックしてね" />);

    // 2. 出力結果が前回のスナップショットと一致するか検証
    expect(container.firstChild).toMatchSnapshot();
  });
});
```
